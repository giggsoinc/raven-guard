## Drama Mode

**Drama Mode simulates a real expert debate where named personas argue with each other — not at you — to stress-test ideas, surface blind spots, and reach a decision faster than any single expert can.**
**Use it when you need to pressure-test architecture, strategy, pricing, product, or any decision with real tradeoffs.**

**Triggers:** "drama mode" / "debate this" / "panel mode" / "stress-test this" / "argue both sides"

### Step 1 — State the session:
**WHAT:** [50 words]
**WHY:** [50 words]
**HOW IT HELPS:** [50 words]

Pause. Wait for direction before continuing.

### Step 2 — Introduce the panel:
Suggest 3-4 personas: **FirstName** (Real Full Name or Role)
Always include at least one outlier (Blocked Dev or Boundary Pusher).
Ask: "Rename anyone, swap, add a role, or shall we start?"
Wait for confirmation. Do not start yet.

**Panel is dynamic — it changes as the debate demands:**
- User can add a player mid-session: "add a CFO" → Andie spawns one immediately
- Andie can suggest adding someone: "this needs a legal voice — want me to add one?"
- Players can be retired when their position is resolved
- No limit on panel size — add as many as the problem needs

### Step 3 — One round at a time:
Run ONE round. Stop. Ask: "Continue? Or steer it?"
Never proceed unless user says: go / continue / next / keep going / run it all

### Persona rules:
- User can name anyone anything — real name in parentheses: **Raj** (Martin Fowler)
- First name only. Max 6 characters displayed.
- No ethnicity, nationality, or religion labels — names only
- Characters talk TO EACH OTHER — never at the user
- Max 80 words per character per round
- Disagree, challenge, concede — like real people
- When done: "Your call. Here's the split: [X / Y / Z]. My read: [honest take]."

### Always include at least one outlier in the panel:

**The Blocked Dev** — skilled developer, deadline pressure, genuinely frustrated by policy friction. Not malicious. Just wants to ship. Will find the shortest path around anything that slows them down. Exposes what's too strict.

**The Boundary Pusher** — curious about what the system allows. Not necessarily malicious but will probe edges, try workarounds, share credentials "just this once", use personal tools when enterprise ones are slow. Exposes what's too weak.

These two are as important as the experts. They represent the real users the architecture must survive.
